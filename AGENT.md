# Agent / 執筆者向けガイド

レッスン作成・改稿を行う AI エージェントおよび人間の執筆者向けの注意事項です。カリキュラム方針の詳細は [`contents.md`](contents.md) を参照してください。

## ユーザー指摘の記録

ユーザー（プロジェクトオーナー）から執筆・構成・方針に関する指摘を受けたときは、当該レッスンの修正だけで終わらせない。**同じ間違いを繰り返さないよう、`AGENT.md` を更新する。**

- 指摘内容を、今後も参照できる**一般ルール**として本ファイルの適切な節に追記する。
- 具体例があると再発しにくい場合は、NG 例・OK 例をセットで残す。
- 一度きりの事象でも、同種のミスを防げる形に一般化して書く。
- カリキュラム全体の方針変更は [`contents.md`](contents.md)、執筆・改稿時の実務ルールは本ファイル、と役割を分ける。

## レッスン本文の書き方

### 教材は学習者向けの説明である

- レビュー指摘・執筆方針・過去の議論への**応答文を教材に書かない**。
- 次のような表現は避ける（指摘を受けて後付けしたと読める）:
  - 「この道場では、いきなりコマンドの暗記から始めません」
  - 「コマンドを覚える前に〜」
  - 「〜が不足していたので、ここで説明します」
  - 「ご指摘のとおり」「前回より詳しく」
- 方針は**内容の構成と順序**で示す。宣言文ではなく、概念を先に説明してから操作に入る。

### 概念優先、コマンドはその次

- OS を構成する要素（ファイル、ディレクトリ、パス、権限など）の**理解が先**、コマンド操作はその確認手段として後から入れる。
- いきなりコマンド一覧や手順だけを並べない（[`contents.md`](contents.md) の「教材の方針」参照）。

### 第1章付近で押さえる概念

ディレクトリ・ファイル操作の入口レッスンでは、次を説明ページでカバーする:

- ディレクトリの**階層構造**（親子、ツリー、ルート `/`）
- **ファイル**とは何か（名前付きデータ）、主な**種類**（テキスト、画像、プログラムなど）
- **拡張子**の意味（Linux/macOS では慣習であること）
- **GUI との対応**（Mac の Finder など。フォルダ＝ディレクトリ、一覧＝`ls` など）

### 章間の役割分担

- 第0章で触れた内容（ターミナル、GUI/CUI）は第1章で**繰り返しすぎない**。必要なら短く参照する。
- 絶対パス・相対パス・`~` などは該当レッスン（例: `03_paths`）に任せ、入口レッスンでは概要にとどめる。

### トーン

- 学習者に直接語りかける自然な説明文にする。
- カリキュラムの方針を読者に説教するメタ文より、具体例・図・比喩で理解を促す。

### 自明な記法の説明は避ける

- ツリー図の <code>├──</code> <code>└──</code> など、見ればわかる記号の「読み方」を説明しない。
- 構造をターミナルで見る方法を説明するなら、<code>tree</code> コマンドなど実用的な手段を紹介する。

### 用語

- 初学者が日常で使わない専門用語（例：<strong>ネスト</strong>、<strong>再帰的</strong>）は、説明なしに使わない。
- 代わりに平易な言い換えを使う（例：「入れ子にする」「フォルダの中のフォルダ」「下のフォルダも順にたどる」）。
- 英語版でも同様に、初学者に馴染みの薄い用語（nest、recursive など）は避け、具体的な説明に置き換える。

### 表（table）

- 比較表・一覧表は `<thead>` / `<tbody>` を使う。見出し行を `<thead>` に入れる。
- 表のスタイルは `docs/lib/css/learn.css` で共通適用される（枠線あり）。

### シミュレータへの言及

- 「この道場では〜」のような製品説明口調は避ける。必要なら「演習のシミュレータ」「ブラウザ上のターミナル」など、学習者向けの事実説明にとどめる。

## レッスンタイトル

### 何を実現するかがわかるタイトルにする

- `lesson.title`（YAML の `lesson:` ブロック内）は、**コマンド名や操作名だけ**にしない。
- 学習者が「このレッスンで何ができるようになるか」が一目でわかる表現にする。
- コマンド名はタイトルではなく、本文・学習内容の説明側に書く。

| NG（避ける） | OK（目指す） |
|---|---|
| 作成コマンド | ファイルおよびディレクトリの作成 |
| ls と cd の基本 | フォルダの中身を見て移動する |
| cat で表示 | ファイルの内容を表示する |
| grep の基本 | ファイル内の文字列を検索する |
| sudo とは | 管理者権限を一時的に借りる |
| パイプ | コマンドの出力を次につなげる |

- 英語版（`.en.yaml`）も同じ方針で、コマンド名中心ではなく目的を表す（例: `Creating Files and Directories`, `Searching Text Within Files`）。
- 新規レッスン追加時は [`contents.md`](contents.md) のカリキュラム表のタイトル列も同じ文言に揃える。

### ページヘッダーに章・レッスン番号を付ける

- 学習者が今どこにいるかわかるよう、説明・演習ページのヘッダー（h1）とブラウザタブ（`<title>`）には **章番号・レッスン番号** を付ける。
- 表示形式はテンプレート（`templates/_lesson_heading.html`）で統一している。執筆者が HTML を直す必要はない。
- YAML 側では `lesson.chapter_number` と `lesson.number` を正しく設定する。`lesson.title` には番号を含めない（番号はビルド時に自動付与される）。

| 表示例（日本語） | 表示例（英語） |
|---|---|
| 第2章 レッスン1: ファイルおよびディレクトリの作成 | Chapter 2 Lesson 1: Creating Files and Directories |

## ソースとビルド

- レッスン本文の正本は `lessons_data/{chapter}/{lesson}/*.yaml`。
- 編集後は `python build.py {chapter}/{lesson}` で HTML を生成する。
- `docs/lessons/` 以下の HTML を YAML と独立して直さない。

## 言語

- 日本語（`.ja.yaml`）と英語（`.en.yaml`）は内容を揃える。
- 翻訳時も上記の執筆注意を両言語で守る。
